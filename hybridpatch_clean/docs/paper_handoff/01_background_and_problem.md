# 01 · 背景与问题（第一章素材）

## 1. 任务背景：长程往返编辑

大模型正在被大量用于"代理式"文档维护：接到指令、修改文档、再接指令、再修改——同一份文档经过模型的手几十次。这类使用方式对模型提出了一个与单次问答完全不同的要求：**没被要求改动的内容必须原样活下来**。

Microsoft 的 DELEGATE-52 基准把这个要求变成了可测量的实验：

- 每个样本是一份（或一组）**专业领域文档**——会计账本、DNS zone 文件、蛋白质 PDB、乐谱、YARA 恶意软件检测规则、LaTeX、Dockerfile 等 52 个领域；
- LLM 先按指令做一次**结构性编辑**（forward，例如"把账本按人拆成多个文件"），再按指令**撤销**（backward，"合并回原来的单一账本"）；
- forward + backward 称为一个 round trip（RT），**链式重复 10 轮**——每一轮的输入是上一轮的输出，错误会累积；
- 每轮 backward 结束后，用**领域专用评估器**（解析账本、解析 PDB、解析乐谱……）把文档与原始版本对比，得到 0–1 的重建分数 **RS**。

10 轮之后 RS 还剩多少，直接反映模型在长程维护中"不该动的东西动没动"。

## 2. FullRewrite 的失败模式

官方基准的默认交互方式是**全文重写**（FullRewrite，下称 FR）：每一步把全部文件的完整新内容重新生成一遍。这带来两类结构性风险：

1. **静默污染**。每一步，文档的每个字节都要经过模型重新"打字"一遍。模型只要在任何与任务无关的位置产生一点漂移——丢一行记录、改一个数值、格式重排——评估器就会扣分，而且这种损伤会作为下一轮的输入被继续放大。10 轮下来 RS 逐轮衰减甚至断崖归零是常见形态。
2. **全量输出的暴露面**。输出长度与文档大小成正比，与编辑量无关。改一行也要重写全文，截断、格式崩坏、漏文件的概率全部随文档变大而上升。

关键点：这不是提示工程能修的问题——只要"保留"依赖模型逐字复述，复述错误就是概率事件；轮数足够多时它必然发生。

## 3. 为什么选补丁路线

补丁（patch）是对抗静默污染的**结构性手段**：模型只声明"要改什么"，其余内容根本不经过模型的手，由确定性执行器原样复制。保留不再是概率事件，而是代码保证。

这个方向上我们的核心可验证声明是一个不变量：**preservation_violations ≡ 0**——未被任何编辑操作声明的内容若被改动，即执行器 bug。整个方法只要这一条字节级承诺成立，"不该动的不会动"就从祈祷变成了合同。

## 4. 前期探索的教训：协议本身会成为负担

在收敛到本文方案之前，我们先做了一条"纯细粒度补丁"路线（内部代号 AnchorPatch），经历了七十多个候选迭代，结论是否定的，但教训直接塑造了最终设计：

> **逐块、逐槽、逐操作枚举的大协议本身成为模型的认知负担。** 模型要同时完成任务理解、全局规划、协议编写、多文件协调、格式维护——这个负担在不少任务里接近甚至超过全文重写。配对实验里纯补丁方法与 FR 在 RS 上统计不可区分：补丁把"复述错误"消灭了，却把等量的错误预算转移到了"协议编写错误"上。

由此得出的判断：**协议救不了协议**。解法不是把补丁协议做得更精细，而是承认不同任务形态需要不同的表达方式——局部小改动用精确补丁最省力；全文件格式转换硬用补丁表达反而是负担，此时"声明边界内的重写"才是对的。这就是 HybridPatch（混合补丁）名字的由来：**多条编辑路径 + 一条显式的有界重写路径**，按任务形态路由，而不是一个协议包打天下。

## 5. 方法概览（详见 02_design.md）

HybridPatch 的单步交互：模型输出**一个 JSON 信封** `{protocol, plan, action}`——`plan` 是计划单（改哪些文件、哪些只读、要产出什么），`action` 是**四选一**的编辑路径：

| 任务形态 | 路径 |
|---|---|
| 少量精确编辑 | `local_patch`（锚点定位的替换/删除/插入） |
| 大量同型修改 | `bulk_patch`（字面全局替换/整行删除） |
| 块搬运/分发 | `dsl_rules`（声明式块复制/分发规则） |
| 整文件生成/转换 | `bounded_rewrite`（**声明过的**文件的整体重写） |

确定性执行器应用信封：声明的部分按指令执行，未声明的部分逐字节保留。任何编辑指令无法安全应用（锚点找不到、歧义、越权改只读文件）都**显式拒绝并记失败**，绝不静默兜底。

## 6. 实验设计的公平性框架（一句话版）

HP 与 FR 在**同一 seeded 任务序列**上逐步配对比较（同样本、同轮次、同编辑指令），主指标是逐步配对的 backward RS 差值；方法的全部失败（无效输出、漏文件、编辑被拒）都按原样计入该方法成绩，不重试、不挑样本。

## 7. 贡献点（供第一章结尾）

1. **方法**：HybridPatch——计划单 + 四路径路由 + 确定性执行的受约束写入协议，把"未声明内容的保留"从模型行为变成执行器保证（不变量 preservation_violations ≡ 0）。
2. **协议设计经验**：给出纯细粒度补丁协议失败的实证归因（协议编写负担吞掉保留收益），论证"按任务形态路由 + 显式有界重写路径"的必要性；以及一组可复用的协议设计原则（减负优先 / 写入有界 / 路径显式 / 失败显式，见 02_design.md §1）。
3. **可审计的实验框架**：配对 relay 实验 + 诚实门（从原始 LLM 响应独立重算全部分数）+ 协议版本门控（旧实验永远可字节级重放），保证报告的每个数字可被第三方复现。
4. **实证结果**（初步，待重跑定稿）：在 20 样本 × 10 轮 × 双臂的配对实验中 HP 整体优于 FR，见 03_results_preview.md。
