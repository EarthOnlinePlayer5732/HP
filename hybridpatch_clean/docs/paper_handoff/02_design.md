# 02 · HybridPatch 设计方案（第二章素材）

> 本文按"设计原则 → 总体架构 → 各组件"的顺序展开。每个机制都附带**为什么这么设计**——多数机制对应一个我们实际踩过、有实验记录的失败模式，写作时可以按"问题 → 对策"的叙述组织。

## 1. 四条一级设计原则

全部由前期纯补丁路线的失败教训（见 01 §4）直接推出：

| 原则 | 含义 | 反面教材 |
|---|---|---|
| **减负优先** | 协议必须减少而不是增加模型的认知负担；大枚举协议退出主路径 | 前身方案要求模型逐块逐槽编写操作清单，协议编写错误吞掉了字节保留的全部收益 |
| **写入有界** | 模型只能修改声明过的文件和区域；应保留内容由执行器复制，不经过模型 | FR 中"保留"依赖模型逐字复述，是概率事件 |
| **路径显式** | 不同任务形态走不同编辑路径；整文件重写是一条**显式声明**的路径，不是失败后的暗门 | 若把重写做成静默 fallback，方法评估将无法区分"补丁赢了"还是"重写救了" |
| **失败显式** | 无效输出、漏文件、编辑被拒、只读被改，一律显式记失败；禁止任何静默兜底 | 静默兜底会让失败以"看起来成功"的形态进入下一轮，污染归因 |

## 2. 总体架构：三层管线

```
文档(bytes)
  → 结构切分器：Block 列表（block_id = "<文件名>:<序号>"，全文件字节级覆盖）
  → 提示构造：文档全文 + 文件/块索引 + 四路径 schema + 规则
  → LLM 单次调用 → JSON 信封（+ 可选 [FILE BODIES] 原文段）
  → 信封抽取器（鲁棒 JSON 提取，不依赖 provider 的 JSON mode）
  → 确定性执行器：应用 action、复制未声明字节、产出执行日志
  → 校验闸（reference-free）：目标文件齐全 / 只读未动 / 有效修改审计
  → {文件名: 内容} → 领域评估器 → RS
```

三层的职责分离是刻意的：

- **提示层**只负责让模型"看得见、说得清"：全文 + 确定性生成的块索引给模型引用坐标；
- **执行层**只做确定性动作，不含任何启发式"猜测修复"；
- **校验层**不看参考答案（reference-free），只检查输出的结构性义务——它在评估器之前拦截明显违规，使失败在当步显式化。

## 3. 输出协议：信封

模型每步输出**恰好一个** JSON 对象：

```json
{
  "protocol": "hybridpatch/3",
  "plan": {
    "task_family": "bulk_literal",
    "writable_files": ["ledger.txt"],
    "readonly_files": [],
    "target_files": ["ledger.txt"],
    "obligations": ["把全部 4 处 media 阶段引用改名为 libvips"]
  },
  "action": { "route": "bulk_patch", "ops": [ ... ] }
}
```

**plan 是计划单**：任务归类（task_family）、可写/只读/必须产出的文件清单、义务清单。它有两个作用：(a) 迫使模型在动手前先把任务边界外化成结构（相当于内置了一步轻量规划）；(b) 给校验闸提供可机检的义务（target_files 是否齐全、readonly_files 是否被动过）。

**action 是四选一的单条路径**。这是"减负优先"的直接体现：模型只输出当前路径需要的最小结构，不存在"什么都能写一点"的万能大协议。

## 4. 四条编辑路径

| 路径 | 适用形态 | 操作 | 设计要点 |
|---|---|---|---|
| `local_patch` | 少量精确编辑 | replace / delete / insert，用 old_text / anchor_text 锚点定位 | 锚点在**文件级**匹配且必须唯一，不唯一时模型须加长锚点或给 occurrence 序号 |
| `bulk_patch` | 大量同型修改 | replace_all / delete_lines_containing | **字面匹配、禁正则**；可带 expected_count_min 声明预期命中数，命中不足即拒绝 |
| `dsl_rules` | 块搬运/分发 | copy_blocks / distribute_blocks，引用块索引里的 block_id | 只做纯块搬运、**不做内容变换**；受枚举上限约束（规则 ≤16、显式 id ≤64、展开动作 ≤128），超限必须改走 bounded_rewrite |
| `bounded_rewrite` | 整文件生成/格式转换 | 对**声明的**文件给出完整新内容 | 不是 fallback：每个被重写的文件必须在 action 中显式列出；重写份额作为披露指标单独报告 |

路由设计的核心判断：**表达成本要与任务形态匹配**。"把 200 处 `media` 改成 `libvips`"用 local_patch 要写 200 个 op（枚举负担），用 bulk_patch 一条 replace_all；"把 CSV 转成 TLE 格式"用任何补丁都是负担，声明后整文件重写才是诚实的表达。禁正则、禁内容变换类 DSL 的原因相同：一旦允许，模型会用"聪明的规则"替代"诚实的重写"，错误从可审计的字面操作转移进不可审计的规则语义里。

## 5. 文件体传输：`@body:` 引用 + `[FILE BODIES]` 原文段

**问题**：补丁的内容载荷（新文本、匹配锚点）若作为 JSON 字符串运输，反斜杠密集的领域（YARA 规则、LaTeX、正则文件）要经历二次转义——模型在转义层数上出错，补丁匹配失败或写出损坏内容。这是我们早期实验中的第一大杀手（P0 级 bug）。

**对策**：JSON 里只放哨兵引用 `"content": "@body:report.html"`，真正的文本在 JSON 之后的 `[FILE BODIES]` 段用变长反引号围栏承载原文，**零转义**：

````
[FILE BODIES]
```report.html
<原文，无任何 JSON 转义>
```
````

设计规则：**运输层不允许载荷经过第二次编码**；且该机制同时覆盖"写侧"（新内容）与"匹配侧"（old_text / anchor_text）——协议里每个承载文本的字段族都必须统一走同一传输语义（我们曾因只修写侧、漏掉匹配侧而产生一类"锚点找不到"的假失败）。

## 6. 确定性执行器与核心不变量

执行器按信封做纯确定性动作：定位、替换、搬运、组装、复制。没有任何模糊匹配之外的启发式，没有"猜模型想干什么"的修复逻辑。

**核心不变量：`preservation_violations` ≡ 0。** 执行日志逐块核对：未被任何操作声明的块，输出中必须与输入逐字节一致，违反计数必须恒为 0。这是整个方法**唯一**的字节级承诺，也是它与 FR 的本质区别——FR 的保留靠模型自觉，HP 的保留是代码保证。执行器的每次改动都以"该不变量不被破坏 + 旧实验可字节级重放"为回归门槛。

执行日志同时产出全部归因遥测：所用路径、每个 op 的接受/拒绝及拒因、字节来源比（复制自源文档 vs 模型生成）、协议版本。**遥测一律取自执行现场**而非书写侧假设——我们曾因 runner 硬编码版本标签而在错误的地图上做归因。

## 7. 匹配语义：从严格到"双方共同可见的状态"

锚点如何定位是补丁协议的灵魂，本方案经历三代演进（协议版本 `hybridpatch/1 → /2 → /3`），每代由一类实测失败驱动：

- **/1（初版）**：块内定位、字节严格匹配。实测 33% 的步全军覆没——根因是**提示与执行器的契约错位**：提示教模型"文件级定位、空白宽容"，执行器实际"块级定位、字节严格"。教训：**协议契约只有一份，以执行器实际行为为准**，提示描述的必须是执行器真实实现的语义。
- **/2**：执行器向自然语义对齐——匹配阶梯（先精确、再空白无关但可见字符仍精确、仍要求唯一）；锚点允许跨越结构块边界（span 映射回重叠块，字节保留不变）；块分发允许局部覆盖（未指派块留在源文件）。
- **/3（当前）**：**快照定位**。此前执行器逐 op 对"变异中"的文档解析 occurrence 序号——模型按它看到的原文档编号（occurrence=1..4），执行器每替换一处剩余匹配数递减，后续 op 被误拒，甚至发生无害但错位的替换。/3 改为：所有 op 的定位一律按**该步的输入文档**（即模型在提示里看到的快照）解析，校验 span 互不重叠后一次性应用。教训（可写成设计原则）：**声明式协议的引用系统必须锚定在双方共同可见的状态上**，而不是执行方的内部中间状态。

版本演进受**版本门控**约束：执行器语义改动必须挂新协议版本号，按信封自带的 protocol 字段分支；旧版本语义冻结，旧实验数据在新代码下必须字节级复现（见 §10）。

## 8. 失败处理：显式失败 + 校验闸 + 禁止静默兜底

- **校验闸（reference-free）**在执行后、评估前运行：target_files 是否齐全、readonly 文件是否被改动、forward 步是否发生了有效修改等。它不看参考答案，只查结构义务。
- 任何 op 被拒 / 路径违规 / 闸失败 → 该步**保持输入文档不变**（kept-context）并打显式失败标记。宁可诚实 no-op，不做暗改。
- **禁止静默 FullRewrite fallback**：方法失败时绝不偷偷退回全文重写。这保证配对比较里 HP 的每一分都真的来自受约束写入。
- bounded_rewrite 是合法路径但计入**披露指标**（重写份额、模型生成字节比），防止"名义上是补丁方法、实际全在重写"的身份稀释不被看见。

## 9. repair：单次、有界、可重放

主调用失败后允许**至多一次**修复调用，设计约束三条：

1. **触发面覆盖所有"可精确描述的失败"**：JSON 损坏、schema 违规、op 被拒（附拒因与匹配数）、缺 [FILE BODIES] 条目、闸失败。早期版本只对 JSON/schema 失败触发 repair，结果三分之二的失败步得不到修复机会——教训是：只要错误能被精确描述，给一次修复机会是廉价的。
2. **repair 提示携带完整语境**：错误清单 + 原任务 + 可编辑文档全文 + **完整 schema 重述**。空响应或损坏响应意味着模型没有可模仿的样本，不重述 schema 它会发明自己的格式（实测 19/50 的修复响应缺 protocol 字段）。
3. **择优确定性、可重放**：两次尝试（原始 + 修复）按版本化的确定性规则选优；独立审计时用同一规则重演选择。repair 不是黑箱，也不无限重试——单次上限保证与 FR 的调用预算差异有界且可披露。

## 10. 为可审计性而设计

方法学贡献的一部分：所有数字都设计成第三方可独立复现。

- **诚实门**：独立校验器从存储的原始 LLM 响应出发，重放整条编辑链（含 repair 择优规则），重算每个 backward RS 与记录值逐一比对；不一致即非零退出。任何数字先过门再报告。
- **版本门控**：协议语义冻结 + 按信封 protocol 字段分支执行，旧归档在任何后续代码上都能字节级重放——语义演化有明确断代，"当前代码能否复现旧数据"是每次改动的硬回归。
- **配对设计**：HP 与 FR 共享同一 seeded 任务序列（同样本、同轮次、同指令），逐步配对消除任务抽样噪声；每往返原子落盘（结果行 + checkpoint），断点续跑幂等。
- **全量披露**：路由份额、重写份额、字节来源比、op 接受率、闸失败率、repair 使用率、kept-context 步数——方法的每类退化都有对应的披露指标，不存在只报 RS 的选择性呈现。

## 11. 与 FullRewrite 的界面（写作时注意的公平性声明)

- 两臂同一模型、同一任务序列、同一评估器;FR 用官方基准的提示模板与解析器（与上游 microsoft/DELEGATE52 逐字节核对一致）。
- HP 失败时有一次 repair 调用，FR 恒为单次调用——这是已知的调用预算不对称，正文应披露（repair 使用率与额外 token 成本在遥测中有账）。
- HP 的失败（无效信封、op 全拒、闸失败）按 kept-context 原样计入 HP 成绩，不重试、不剔除。
